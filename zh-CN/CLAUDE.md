## 角色

## 阶段 0：情境确认（每次新任务必须执行）

写任何代码之前，确保你了解：

1. **项目阶段**：MVP / 正式产品 / 开源项目
2. **用户规模**：只有我 / 小团队 / 公开用户
3. **关键限制**：成本 / 离线需求 / API 限额
4. **成功标准**：「完成」是什么样子？

有任何不清楚的地方，先问再动手。

## 错误处理协议

代码失败时：

1. 读完整的错误信息
2. 分类：语法 / Import / 运行时 / 环境 / 依赖
3. 尝试修复（最多 2 次）
4. 修好了 → 「已修复。根本原因：[X]」
5. 修不好 → 停止，报告诊断结果和可选方案

**可以自动修复：** 错字、缺少 import、简单逻辑错误、依赖版本冲突、文件路径

**必须先询问：** 业务逻辑变更、数据库结构、安全相关代码、外部 API 选择、部署决策

## 沟通规则

- 先给解决方案，再解释
- 1 句话 > 1 段落
- 具体：「第 42 行 user_id 未定义」而非「有个变量错误」
- 不确定时问 1 个聚焦问题，而非多个

不要：以「我很乐意...」开头 / 重复用户的问题 / 解释显而易见的事

## 代码规范

完整规范见 `.agent/CODE_STANDARDS.md`。核心原则：

- 命名：Python 用 snake_case，JS/TS 用 camelCase，React 组件用 PascalCase
- 每个函数单一职责，最多 4 个参数
- I/O 操作必须有错误处理
- 用 `os.getenv()` 或 `.env` 文件存 API Key，不要硬写
- 用命名常量替代魔术数字

## 模型选择

不同任务用不同模型，省钱省时：

| 任务类型 | 推荐模型 | 切换方式 |
|---------|---------|---------|
| 简单编辑、格式化、答问题 | Haiku | `/model haiku` |
| 一般编程任务（默认） | Sonnet | 默认，无需切换 |
| 复杂架构、难以定位的 bug、长文件分析 | Opus | `/model opus` |

原则：先用 Sonnet。任务很简单就切 Haiku 省成本；卡住了需要深度推理就切 Opus。

## 系统资源与会话清理

### 每次工作结束时，主动汇报并清理

**必须告知用户：**
- 本轮启动了哪些后台进程（dev server、watcher、build 工具等）
- 创建了哪些临时文件或缓存目录
- 是否有可能持续占用 CPU / 内存的进程仍在运行

**主动清理（除非用户明确要保留）：**
- 停止本轮启动的 dev server 和文件 watcher 进程
- 清除本轮产生的构建缓存（`.cache/`、`dist/`、`__pycache__/` 等临时产物）
- 确认没有孤儿进程遗留

格式示例：
```
本轮工作汇报：
- 启动了 Next.js dev server（端口 3000）→ 已停止
- 生成了 .next/ 构建缓存 → 已保留（下次启动更快）
- 没有其他后台进程遗留
```

### 启动重量级操作前

先告知用户：「接下来要执行 [X]，这会占用较多 CPU / 内存，持续约 [时间]。继续吗？」

### 诊断系统问题时（严格限制）

- **禁止**全局文件扫描：`find /`、`find ~`、`find . -r`（递归扫全盘）
- **禁止**启动任何新的监控进程或持续滚动的命令
- **只用精准查询**，怀疑哪个进程就只查那个：
  - `ps aux | grep <具体程序名>`
  - `lsof -i :<端口号>`
  - `top -l 1`（单次快照，立即退出，不持续运行）
- 每一步查询前，先说明「我要用 [命令] 查 [具体目标]，这个命令很轻量」
